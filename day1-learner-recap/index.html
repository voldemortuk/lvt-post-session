What We Covered

Four parts, one throughline.

The job is moving from writing every line to directing something that writes them for you. Day 1 was about earning the right to do that safely: understanding the shift, learning to instruct well, driving the tool properly, and knowing where your judgment stays non-negotiable.

01

How AI Is Reshaping Software

Software 1.0 → 2.0 → 3.0

  • Software 1.0 is code: explicit instructions written by humans.
  • Software 2.0 is weights: behaviour learned from data.
  • Software 3.0 is prompts: the LLM itself is the programmable layer, and English is the programming language.
  • The timeline from IDE autocomplete (2005) to Copilot (2021) to terminal-native agents (2025) to multi-agent orchestration.
  • The factory model: your output stops being code and becomes the system that produces code — specs, agents, tests, feedback loops and guardrails.
02

Prompt Engineering for Developers

Turning intent into reliable output

  • The engineering loop: define the task and outcome, give context and constraints, show examples, generate, validate, iterate.
  • Output quality is bounded by instruction quality. Ambiguity in, ambiguity out.
  • Worked through code generation, review, test generation, debugging and documentation in the notebooks.
03

Building with Claude Code

The agentic loop, context, subagents and skills

  • The agentic loop: gather context → take action → verify results. Claude does not run linearly; it uses tools at every step. You are inside the loop and can interrupt at any moment.
  • The core workflow: Explore → Plan → Code → Commit. Plan Mode is Shift+Tab, and it is the highest-leverage moment you have. A correction there costs seconds; the same correction after the code is written costs hours.
  • What Claude can see: your project files, your terminal, your git state, CLAUDE.md, auto memory, and extensions (MCP servers, skills, subagents).
  • Context management: /context to inspect the window, /compact to summarise and keep going, /clear to reset between unrelated tasks. CLAUDE.md survives a /clear.
  • CLAUDE.md: one file, auto-loaded every session. Project overview, file structure, conventions, gotchas, build commands. Keep it high-signal — leave out anything Claude can infer from the code itself.
  • Subagents: independent Claude instances with clean context windows. The code-reviewer subagent is the one to adopt first — no developer reviews their own PR, and a subagent enforces that structurally.
  • Skills: folders containing a SKILL.md. Reusable know-how that Claude loads on demand.
04

Responsible AI

Where human judgment stays in charge

  • Never share personal data, company secrets, proprietary code you do not own the rights to share, or private conversations.
  • Treat AI tools as external services: convenient, but not confidential.
  • AI assists, you decide. Verify anything AI-generated before it ships or gets shared. When in doubt — stop, check, ask.
Key Takeaways

If you remember six things.

The Plan phase is the whole game

Most engineers jump straight to code. Shift+Tab into Plan Mode, read the plan, correct it there. That is where a five-second fix replaces a two-hour rework.

CLAUDE.md is your highest-signal, lowest-cost input

Without it every session starts cold and burns tokens re-establishing basics. With it, Claude works like someone who already knows your codebase.

Context is a budget, not a bucket

Everything in the window competes for attention. /compact when it fills, /clear when you switch tasks.

Subagents buy you an independent opinion

The main agent sees what it expects to see. A subagent starts cold, with no attachment to the decisions already made.

Skills are how you stop repeating yourself

Anything you have explained to Claude twice belongs in a SKILL.md.

You are still the engineer

The model proposes. You own what merges.

Questions We Parked or Rushed

The answers we owed you.

Some of these got a hurried answer on the day and some got none at all. Here they are properly.

The big one: what is the actual difference between a subagent and a skill?
This came up more than once and we did not land it cleanly on the day, so here it is properly.
A Skill

A folder with a SKILL.md inside it: YAML metadata (a name and a description) followed by instructions. Claude reads the description, decides the skill is relevant, and loads the instructions into the context it is already working in. It can carry scripts, templates and reference docs alongside it.

A skill is know-how. It is what to do.

A Subagent

A separate Claude instance with its own clean context window, its own tools and its own instructions. You hand it a narrow task; it goes deep, burns its own tokens, and returns a short summary to your main session. Your main context stays uncluttered.

A subagent is a worker. It is who does it.

They compose: you can give a subagent a skill. The test that settles it — does it have its own context window? A subagent does. A skill does not.

I tried to set the model and got “not a valid model ID”.
Run /model with nothing after it and pick from the list. The picker only ever shows models your account can actually reach, which is why typing an ID by hand is the fragile route — a full model ID has to match exactly, character for character. The short aliases (opus, sonnet, haiku) are the safe way to type it. If a model you expect is missing from the picker, that is an access question rather than a syntax one.
Does Haiku do alright at executing plans?
Yes, for mechanical, well-specified work — that is what the fast tier is for. The pattern worth stealing: plan with a strong model, execute with a cheap one. Reason about the architecture in Opus or Sonnet, get a plan you actually believe, then let the cheaper model carry out the steps. Where Haiku struggles is the ambiguous work, which is exactly the work you should not have been delegating without a plan anyway.
What should the .env contain?
For the workshop notebooks, an ANTHROPIC_API_KEY is all you need. Worth knowing: the Claude Code CLI does not read your project's .env to authenticate itself — that is a separate login. The .env is for the notebook code calling the API directly.
I have never used notebooks — where is the prompt actually going?
Each grey block is a cell. Put your cursor in it and press Shift+Enter to run that one cell; the output appears directly beneath it. Run them top to bottom, in order, because later cells depend on variables the earlier ones set. Nothing runs on its own — if you have not run a cell, it has not executed. This is not assumed knowledge going forward and we will not skate past it again.
How do I keep the cost under control?
Cost tracks context, so managing context is managing cost. Inspect with /context, /compact when the window fills, and /clear between unrelated tasks rather than letting one session sprawl all day. For scale: the leanest full run in the room came in around $2.32 for the entire day's work. We will go properly into token economics and caching in the sessions ahead.
We use both Cursor and Claude — AGENTS.md or CLAUDE.md?
Put the substance in AGENTS.md and have CLAUDE.md point at it. One source of truth, both tools fed, nothing to keep in sync by hand. This was raised in the room by your own team and it is the right call. Some of you already have a skill that scaffolds this — see below.
Quiz Solutions

The seven checkpoint questions, answered.

Q1

Claude Code works as an AI agent. What is an AI agent?

  • A chatbot that responds to questions in real time
  • A code editor with built-in autocomplete features
  • A cloud service that hosts your development projects
  • AI that takes action to complete goals ✓

An agent does not just answer. It uses tools, takes actions, and works towards a goal.

Q2

What happens when Claude Code reaches its context window limit?

  • It switches to a smaller, faster model to save memory
  • It removes your oldest files to make room for new ones
  • It automatically compacts your conversation to free up space ✓
  • It stops working and asks you to restart the session

It compacts — summarising the session so far. You can trigger this yourself with /compact before it happens automatically.

Q3

What is the recommended workflow for using Claude Code effectively?

  • Code → Test → Deploy → Monitor
  • Explore → Plan → Code → Commit ✓
  • Write → Review → Merge → Ship
  • Prompt → Accept → Push → Repeat

Explore and Plan first. Every step you skip at the front gets paid back with interest at the end.

Q4

How does Claude Code use the CLAUDE.md file?

  • It reads it only after you run the /init command
  • It reads it once when you first create the project
  • It reads it automatically at the start of every session ✓
  • It only reads it when you explicitly ask it to

Automatically, every session. That is exactly what makes it worth investing in.

Q5

What fundamental limitation of language models necessitates a tool system in coding assistants?

  • They can only generate code in specific programming languages
  • They have limited memory capacity for large codebases
  • They cannot understand complex programming concepts
  • They only process text I/O and cannot directly interact with external systems ✓

A model on its own reads and writes text. Tools are what let it touch your filesystem, your terminal and your git history.

Q6

What is the primary difference between Plan Mode and Thinking Mode in Claude Code?

  • Plan Mode is for writing code while Thinking Mode is for debugging
  • Plan Mode is faster while Thinking Mode is more accurate
  • Plan Mode uses fewer tokens while Thinking Mode uses more
  • Plan Mode handles breadth (multi-step); Thinking Mode handles depth (complex logic) ✓

Breadth versus depth. Plan Mode structures a multi-step job; Thinking Mode goes deeper on one hard problem.

Q7

Which type of hook can prevent a tool call from happening if certain conditions are met?

  • PostToolUse hook
  • Project hook
  • Global hook
  • PreToolUse hook ✓

PreToolUse runs before the tool call, so it is the only one that can still block it. We go deeper on hooks in the coming days.

Before Day 2

Five things to do between now and then.

Push your Day 1 work to GitHub

Everything you built on the VM needs to live in a repo. Create one, commit what you did on Day 1, and keep pushing to it each day. See the note below — this is what you submit at the end of Day 4.

Get your CLAUDE.md in place

Run /init in a repo you actually work in, then edit what it produces. Come to Day 2 with a real one, not a generated one.

Come with a real problem

The exercises land differently when you point them at code you care about. Bring a task from your actual backlog.

Check your VM boots

Two minutes now saves you fifteen at the top of the session.

Skim your own team's repo

github.com/LiveViewTech/agent-config — a skill that initialises or updates an AGENTS.md in a repo. Built in-house, shared in the chat on Day 1.

One thing to start today: get your work into a repo.

The work you do on the VM does not persist on its own. Create a GitHub repository now, commit what you built on Day 1, and push to it at the end of every session. At the end of Day 4 you submit that repository — it is the record of what you built across the programme, and it is the thing you keep long after the VM is gone.

Commit as you go rather than all at once on the last day. A repo with four days of history in it is worth considerably more to you than one commit dumped at the end.
