01

AI-Assisted Coding & AI Adoption

What a controlled study reveals about speed, skills, and how you should use AI

  • A new Anthropic study — Shen & Tamkin (2026) (Anthropic writeup · full paper) — ran 52 engineers through a randomized controlled trial: build two features with an unfamiliar async library, with or without AI, then a quiz on the concepts just used.
  • The result: the AI group scored 17% lower on the post-task quiz — about two letter grades (~50% vs ~67%, Cohen's d = 0.738, p = 0.01). The biggest gap was in debugging, exactly the skill you need to catch AI's mistakes. The speed gain was only ~2 minutes and not statistically significant.
  • What actually mattered was HOW people used it. Low scorers (<40%) fully delegated code generation or only used AI to fix errors. High scorers (65%+) generated code then asked follow-up questions, requested explanations alongside code, or asked conceptual questions while coding solo. Cognitive effort — even getting stuck — is what builds mastery.
  • The adoption journey runs in stages: drop the copy-paste chatbot for an agent that reads files and runs programs → reproduce your own past work to learn the tool's limits → kick off end-of-day agents for a warm start tomorrow → outsource the slam dunks, keep the deep work → turn every mistake into a rule in AGENTS.md → eventually, keep a useful agent running in the background.
  • The non-negotiable: verify, test and review everything — treat AI output like a junior developer's, confident but error-prone. Commit often as save points. Customize behaviour with CLAUDE.md and style guides. AI amplifies fundamentals: strong engineers get multiplied, weak foundations get amplified confusion.
02

MCP & Hooks

Connecting Claude Code to your tools, and automating your guardrails

  • MCP (Model Context Protocol) is an open standard connecting Claude Code to file systems, databases, third-party APIs, browsers, custom tools and search engines — think of it as USB-C for AI agents, one connector instead of one cable per device.
  • An MCP server exposes three things: Tools (actions Claude can execute — open a PR, query a database), Resources (data Claude can read — a Notion doc, a log file), and Prompts (reusable templates for common tasks).
  • The call flow: Claude Code (the MCP host) connects via an MCP client → the model picks a tool → the client sends the request over the MCP protocol → the server runs and calls the real API → the result returns to the model. The worked example from the slides: fixing a Jira bug end to end — pull the ticket, read and edit the source file, run the tests, open a GitHub PR, notify the team on Slack.
  • Hooks are user-defined shell commands Claude Code runs automatically at specific lifecycle points — PreToolUse (before a tool call, can block it), PostToolUse (after it completes), Notification, and Stop. Instead of prompting Claude to "always run the linter," you wire it in: edit → hook fires → linter runs. Guaranteed, every time, zero tokens.
  • A hook controls the action three ways: exit 0 allows it, exit 2 blocks it and tells Claude why, or a JSON decision can allow, deny, ask, or rewrite. Configured in settings.json or /hooks, and can be committed so the whole team gets them.
03

Claude Code Plan Mode

Separate thinking from doing

  • Why plan first: assistants are trained to be helpful and start writing code immediately, making silent assumptions about your architecture. At roughly 80% right per decision and ~20 decisions in a typical feature, that's about a 1% chance everything is correct. The fix isn't a cleverer prompt — it's separating thinking from doing.
  • Plan Mode is read-only: Claude can read files, search the codebase, list directories, search the web, spawn research sub-agents, and ask clarifying questions — but it cannot write, edit, or run shell commands. The output is a reviewable plan; nothing is implemented until you approve it.
  • Three modes: Normal (asks before each edit), Auto-Accept (executes without asking — for running an approved plan), and Plan Mode (read-only analysis for complex, multi-file, or unfamiliar work). Enter it with Shift+Tab twice, /plan mid-session, or --permission-mode plan at launch.
  • The core workflow: Explore → Plan → Implement → Commit. The first two phases are the cheapest in tokens and the most valuable in outcome — they prevent solving the wrong problem.
  • The plan is a contract: open it in your editor (Ctrl+G), annotate, delete steps, reorder — Claude picks up every change. Store plans in your repo and they become decision records and ready-made PR descriptions.
  • The one-sentence rule: if you can describe the exact change in one sentence, just do it. Plan when the task touches 3+ files, involves an architectural decision, or the codebase is unfamiliar.
04

Spec-Driven Development

From vibe coding to executable intent

  • Vibe coding — describe a goal, get code back — is fine for quick prototypes but unreliable for serious work. The problem isn't the agent's ability; it's treating a literal-minded pair programmer like a search engine. "Add photo sharing to my app" forces it to guess thousands of unstated requirements, and every handoff between stakeholder needs, requirements, design, implementation and validation is a place intent can leak.
  • The big idea: from "code is the source of truth" to "intent is the source of truth." AI is what makes a specification executable.
  • The lifecycle: Constitution (principles, standards, tech stack — persists across sessions) → Specify (the what and why, no tech stack yet) → Clarify (resolve ambiguity before planning) → Plan (now go technical) → Tasks (break the plan into small, independently testable chunks) → Implement & Validate. The core four everyone shares: Specify → Plan → Tasks → Implement.
  • Small tasks over vague asks: "build authentication" makes the agent guess. "Create a registration endpoint that validates email format" gives it something to verify. Concrete, reviewable tasks the agent can implement and test in isolation — almost TDD for the AI.
  • Where it shines: greenfield (0→1) builds, feature work (N→N+1, its most powerful use), and legacy modernization — recapturing lost intent in a fresh spec.
  • Stay the human in the loop: run "grill me" sessions where the AI interviews you before any code is written, favour vertical slices with failing-tests-first, design deep module interfaces yourself and delegate the internals, and retire old specs so the agent never builds against stale requirements.
Key Takeaways

If you remember six things.

Delegation without engagement costs you the skill

The controlled study found a 17% quiz-score gap for the AI group, with the largest hit landing on debugging — the exact skill you need to catch AI's mistakes.

It's how you use AI, not whether

Fully delegating or using AI only to fix errors is what tanked scores. Asking follow-up questions and requesting explanations alongside code is what kept people learning.

Plan Mode turns 1% odds into a contract

Twenty compounding assumptions at 80% accuracy each is roughly a 1% chance everything is right. Plan Mode makes you review the plan, not the aftermath.

Hooks enforce structurally, not conversationally

A hook runs outside Claude's context window, adds zero tokens, and never depends on Claude interpreting an instruction correctly. "Always run the linter" becomes guaranteed, not hoped-for.

MCP is the one connector for every tool

Tools it can call, resources it can read, prompts it can reuse — one protocol instead of a bespoke integration per service.

Intent is the source of truth

Code is what AI accelerates. The spec is what keeps that acceleration pointed at the right target.

Questions We Parked or Rushed

The answers we owed you.

Some of these got a hurried answer on the day and some got none at all. Here they are properly.

The big one: would a hook built for Claude Code work the same way in Cursor? And why write Day 2's demo hooks in shell script rather than TypeScript or Python?
Two related questions from the same demo, and the honest answer on the day was "I haven't used Cursor a lot" — worth a proper one here.
Claude Code

Runs a pure agentic loop: the model plus a small set of tools (bash, read, write, glob, grep). Hooks are shell commands wired into that loop at specific lifecycle points — PreToolUse, PostToolUse, and so on.

Cursor

Built around repository indexing and retrieval rather than an explicit tool-call loop. It has its own "rules" mechanism, but a rule and a hook aren't the same primitive — nothing here ports over 1:1.

The portable layer across tools isn't hooks — it's files you commit to the repo. A CLAUDE.md/AGENTS.md pair works everywhere because every tool reads plain files. Hooks are Claude Code-specific enforcement; treat them as a Claude Code feature, not a cross-tool standard.

On the shell script choice: it wasn't a technical requirement — a hook can be any executable. Shell was used for the demo because it needs no extra dependency to run and stays short enough to read inline. TypeScript or Python earn their keep once a hook needs real logic — JSON parsing, calling another service, structured error handling — which gets awkward fast in bash.

How much token spend is typical? We're not sure if our spend limits are standard, high, or low.
This one didn't get an answer on the day, so here's the honest one: there isn't a single right number. Spend scales with model tier, how much context you're feeding in, and how many subagents or tool calls a task triggers. A day like Day 2 — MCP, hooks, Plan Mode — typically runs heavier than a plain code-generation day, simply because every tool call and every hook firing adds to the loop. For a concrete read on your own usage, /cost shows session-level token spend and /context shows what's filling the window right now. Whether your specific limits are standard for a team your size is a provisioning question for your account admin — we don't have visibility into your organization's plan from the training side.
If the AI is just going to expand our short 5-point list into a full ~120-line constitution anyway, why not have it work straight from the short version and save the tokens?
A sharp question that didn't get a direct answer on the day — the conversation moved toward other tools instead. Here's the direct one. A five-point list is fine as input, but it's not fine as a working document: it's compressed, and every gap in it forces the agent to guess. Expanding it into a full constitution front-loads the same problem Plan Mode exists to solve — you pay the token cost once, in a form you can actually read and correct, instead of paying a smaller ambiguity tax every time the agent has to reinterpret a terse bullet point mid-generation. It's also reused, not single-use: the constitution persists across every session for the life of the project, so its cost amortizes across every planning and implementation run that reads it — unlike a five-point list the agent would otherwise have to reinterpret from scratch each time.
Are the specs, plans, and tasks documents version-controlled with the rest of the repo, or kept separate?
Yes — and it's worth the full answer rather than the one-line version we gave live. Spec-kit style tooling keeps the constitution, specs, plans and tasks as plain markdown files inside a project folder (commonly .specify/ or specs/) that lives in your normal git history, not a separate system. That means normal code review applies: a diff shows exactly when a spec changed and why, and a PR comment can question a requirement the same way it questions a line of code. The one decision worth making as a team is what happens to a spec once its feature ships — keep it as a permanent decision record, or archive it. Leaving that undecided is the "doc rot" risk flagged in the SDD section: an agent building against a spec nobody retired is building against a stale requirement.
I'm still unclear where prompts stop and agents start.
A fair thing to still be fuzzy on — this one never got picked up live, so here's the clean version. A prompt (like the /format slash command from the Day 2 demo) is a static, reusable piece of text you invoke on demand; it does the same fixed thing every time and has no autonomy of its own — it's just text inserted into your current session. A skill is reusable know-how (a folder with a SKILL.md) that gets loaded into the agent's existing context when its description matches what you're doing — still the same agent, same context window, just handed more instructions mid-task. An agent (or subagent) is different in kind: a separate reasoning loop with its own context window and its own tool access, running autonomously across multiple steps and reporting back a result. The dividing line is autonomy: a prompt has none, a skill borrows the calling agent's, and an agent has its own.
How does MCP avoid tool collisions with intent-based interaction?
Never picked up live, and it's a fair thing to wonder once you've connected more than one MCP server. The short answer: tool names get namespaced by which server they came from, so two servers can each expose a tool without the model confusing them — Claude Code's own tool names follow exactly this pattern (mcp__servername__toolname). Given a request, the model picks among those fully-qualified names by matching your intent to each tool's description — the same mechanism that picks between two skills. Which means the same failure mode also applies: if two tools across different servers are under-described or too similar, selection gets unreliable, the same way two similarly-worded skills do. The practical fix is the same one covered in the skills section — write specific, differentiated tool descriptions, don't rely on the namespace alone to disambiguate intent.

Q1

What best describes Plan Mode?

  • A mode that auto-executes edits without asking
  • A read-only mode: the agent can read, search, and ask — but not change the project ✓
  • A faster model for simple fixes
  • A mode that commits code automatically after each step

Plan Mode is read-only by design. Nothing in your project changes until you approve the plan and switch out of it.

Q2

According to the "one-sentence rule," when should you skip planning?

  • When the task touches many files
  • When you're unfamiliar with the codebase
  • When you can describe the exact change in a single sentence ✓
  • When an architectural decision is involved

If the change is that obvious, planning is overhead. The other three are exactly when you should plan.

Q3

What core shift does spec-driven development represent?

  • From "intent is the source of truth" back to "code is the source of truth"
  • From writing tests first to writing tests last
  • From "code is the source of truth" to "intent is the source of truth" ✓
  • From using agents to avoiding them entirely

The spec becomes the shared, executable artifact both humans and AI build against — not the code itself.

Q4

Why does the spec-driven workflow break work into small, concrete tasks?

  • So they can be implemented and tested in isolation, keeping the agent on track ✓
  • So the developer never has to review the output
  • Because agents can only read one file at a time
  • To make the spec longer and more detailed

Small tasks are reviewable in isolation — the difference between "build authentication" and "create a registration endpoint that validates email format."
