What We Covered

Five parts, one gate.

Day 2 was about directing the tool with discipline. Day 3 was about not trusting it — building the gates a team can actually stand behind before AI-written code ships: catching your own mistakes first, getting judgments that aren't the author's, wiring those reviews into GitHub itself, hunting real vulnerabilities instead of trusting a tool's word for it, and setting permissions that hold without slowing you down.

01

Anti-Slop & Reviewing AI Code

Catching the bugs that survive because the code looks right

  • The day's one idea: an AI first draft that looks idiomatic is the most dangerous kind of code, because tidiness disarms the reviewer — you read the shape, nod, and move on. The shape is what the model gets right; the logic is what it gets wrong.
  • What slop looks like: plausible-but-wrong helpers (an inverted comparison that reads perfectly), copy-paste drift (a status/route/field never adapted), dead branches (conditions that can't be true given the data model), tests that assert nothing, and off-by-one errors. A twelve-line handler can hide four of these at once, all cleared by one vacuous test.
  • The fix is a system, not a harder stare: a scored checklist read for intent in five passes — correctness & logic, copy-paste drift, error handling, tests that assert nothing, and hygiene.
  • /self-verify wires this into the agent itself: deterministic gates first (lint/type-check → tests → build, never present over a failing gate), then a structured self-critique re-reading the diff for logic, drift, error-handling, weak tests, and scope creep, then present the gate results, the critique, and the diff together.
  • Scaffolding vs. gate: line-by-line reading builds your instinct for what slop looks like, but it fades as tooling matures — you do less of it over time. Tests, lint, and contracts are permanent gates: deterministic, tireless, no judgment call required. A green test that asserts nothing is still red; strengthen the assertion and re-run.
02

Adversarial Review: Multi-Hat Thinking

Independent judgments a single reviewer blends away

  • The problem: one reviewer asked for security, design, performance, and correctness all at once tends to look solid and still miss the IDOR. Wearing four hats at once means doing each one poorly.
  • Three hats, one session each, no cross-talk: /hat-security (the attacker's lens — injection, IDOR, SSRF, XSS, hard-coded secrets), /hat-design (the six-months lens — copy-paste drift, wrong boundaries, misleading names, what will hurt to maintain), and /hat-blackbox <url> (contract-only — it may not read the source, only contracts/openapi.yaml).
  • Why separate sessions matter: one shared context blends the conflicts away; three independent vantages stay sharp and are allowed to disagree — the security and design hats will clash over an allowlisted ORDER BY, and that clash is the point, not a bug.
  • The black-box hat can't be talked out of a bug. Because it sees only the contract, never the implementation, it can't anchor to the code's intent or quietly weaken a test to pass — it tried the obvious cross-tenant request the contract promised was blocked, and caught the IDOR that a reviewer reading tidy code never would.
  • Keep a human involved: never ship AI code covered only by AI-written tests. At least one human-authored assertion belongs on anything that matters — the kind of test an agent's own happy-path suite would never think to write.
02.1

GitHub Actions

Running the same review gates automatically, on every PR

  • The hands-on: a self-walkthrough guide (shared live in chat) had learners fork anshajk/claude-security, install the Claude GitHub App, add ANTHROPIC_API_KEY as an Actions secret, enable workflows on the fork, and open a PR — roughly 7–8 people completed it live. The OWASP-specific review only triggers when the change lands inside backend/src/**, frontend/**, or contracts/openapi.yaml.
  • anthropics/claude-code-action@v1 runs Claude Code as a workflow step — your code stays on GitHub's own runners. Two ways to trigger it: interactive (an @claude mention in an issue or PR comment) or automation (a prompt: field firing on pull_request, schedule, or workflow_dispatch).
  • Two separate lanes, two separate runners: the deterministic lane (trust-gate-ci — lint, build, test, a secret scan) is what actually blocks a merge. The judgment lane (claude-code-action@v1 posting hat and OWASP findings as PR comments) only advises — severity never fails a build by design. Want a finding to block? Write the assertion into the deterministic suite; a review comment is a claim, a failing test is proof.
  • Decisions baked into the workflow file: paths: wakes the job only on the vulnerable surface, concurrency with cancel-in-progress means a force-push cancels the stale run, permissions: scopes GITHUB_TOKEN (it only reaches Claude if you pass it in explicitly), and checkout has to run before the action or .claude/skills never lands.
  • The headless trap: in CI there's no human to answer a prompt, so an ask rule effectively means deny — a Bash(curl:*) ask rule silently kills every black-box-hat probe. Fix: delete the ask rule and run --permission-mode dontAsk with an explicit allow instead; bypassPermissions is not the fix, since it survives the trap but also approves every tool, not just the ones you listed.
  • Setup is /install-github-app (repo admin only) — wires the GitHub App and the ANTHROPIC_API_KEY secret, scoped to Contents, Issues, and Pull requests (read and write). CODEOWNERS on .claude/** and .github/** means changing the rules themselves needs a human review too.
03

Security: Secure-by-Default & OWASP

Catching vulnerabilities while the agent writes, not after

  • Five vulnerability classes showed up in the day's Command Center build: injection (user text glued into a query runs as code), broken access control / IDOR (a missing tenant filter), SSRF (server fetches a user-supplied URL), stored XSS (untrusted input rendered as HTML/JS), and secrets/misconfig. One idea to hold onto: authenticated ≠ authorized — logging in proves who you are, not that you're allowed to see this row.
  • A tool flag is a claim, not a proof. Every OWASP pass produces three buckets: caught (real, fix it and record the decision), false positive (dismiss it in writing — an allowlist-guarded ORDER BY is reviewed-safe, not flagged), and caught-but-unproven (a human cross-tenant test proves it either way).
  • The owasp-review skill lives at .claude/skills/owasp-review/ so every engineer inherits identical checks, applying secure patterns as the agent writes. It runs a fixed five-category method in order: injection, IDOR, SSRF, XSS, secrets & config — each finding reported as severity, what, exploit, fix, and confidence.
  • An independent security-reviewer subagent (.claude/agents/security-reviewer.md, tools scoped to Read, Grep, Glob, Bash(git diff:*)) runs a second, separate OWASP pass on a diff — judgment, distinct from the skill's automatic coverage. Paired with a deterministic secret-scan.sh hook on PreToolUse that blocks a planted secret outright: two different gate types, and you need both.
  • /triage turns a flagged finding into a written verdict: locate the exact file:line, trace input to sink (is there a tenant predicate for authz?), reach a verdict — real or false-positive — and record it. A false positive dismissed without a written reason is indistinguishable from a real bug ignored.
  • The other OWASP list — AST10: the web Top 10 is about bugs in your app; the Agentic Skills Top 10 is about the risk of the skills and plugins you install into your agent, since a skill runs with your agent's privileges. The lethal trifecta: a skill is dangerous once it can read private data, see untrusted content, and reach the outside network, all three at once. That's exactly why this bootcamp ships its own vetted, self-contained owasp-review skill — no marketplace install, nothing pulled from the network.
04

Permissions Without Constant Interruptions

A committed team standard, not a per-session judgment call

  • How Claude Code decides: every tool call is checked against three lists in .claude/settings.json — deny (never allowed), ask (prompt every time), allow (run without asking) — in that order, first match wins. How specific a rule is doesn't matter; a broad deny Bash(aws *) beats a narrow allow Bash(aws s3 ls). And rules are enforced by Claude Code, not the AI — "don't do X" in a prompt or CLAUDE.md is a request, not a fence.
  • Sort every action into allow / ask / deny: allow what's safe, frequent, and easy to undo (reads, tests, lint, curl localhost); ask before anything powerful or external (git push, WebFetch, edits to .claude/); deny what's never acceptable regardless of context (.env reads, the cloud metadata IP, rm -rf, force-push). The physical-action rule: never auto-allow anything irreversible or with a real-world effect.
  • A rule that looks safe can still be wide open. An ask: Bash(curl example.com *) rule doesn't stop curl -X GET example.com, a different host, a redirect, or a variable holding the URL — fix is to deny curl/wget outright and allow only WebFetch(domain:example.com). Likewise allow: Bash(npx *) also allows npx rimraf .; name the exact inner command instead, one rule per tool.
  • Where rules live: User (~/.claude/, just you everywhere), Project (.claude/settings.json, committed, everyone on the repo), Local (.claude/settings.local.json, gitignored, just you on this repo), and Managed (pushed by IT/DevOps, the whole company, can't be overridden). When they disagree, Managed > Local > Project > User — but the deny lists always merge together regardless of level.
  • One rule has a blind spot, so layer them: a permission rule stops the agent's own tools and known commands, but not a script the agent writes that opens the file itself. Defense in depth is three layers — the permission rule, a PreToolUse hook (your own detection script), and the OS sandbox — each catching what the one before it misses. Gate the gate: put CODEOWNERS on .claude/** too, so changing the rules themselves needs a human review.
Key Takeaways

If you remember six things.

Tidiness disarms the reviewer

An AI draft that looks idiomatic is the most dangerous kind of code — you read the shape, nod, and move on. The shape is what the model gets right; the logic is what it gets wrong.

Independent hats beat one blended pass

A single reviewer asked for everything at once tends to miss the IDOR. The value of the panel isn't any one hat — it's the deltas between them.

Only the deterministic lane can block a merge

GitHub Actions runs Claude's review automatically on every PR, but severity never fails a build by design. Want a finding to block? Write it as an assertion in the test suite, not a comment.

A tool flag is a claim, not a proof

Every OWASP pass produces caught, false-positive, and caught-but-unproven buckets. Triage turns a flag into a written verdict — a dismissal with no reason is indistinguishable from a bug ignored.

Deny, then ask, then allow — first match wins

Specificity doesn't reorder the check. A broad deny always beats a narrow allow, and every level's deny rules merge together regardless of where they're set.

No single gate is enough

A permission rule stops the obvious path; a hook catches what slips past an allow rule; a sandbox survives a script the agent writes itself. Defense in depth means layering all three.

Questions We Parked or Rushed

The answers we owed you.

Some of these got a hurried answer on the day and some got none at all. Here they are properly.

I got behind and missed how the agent was kicked off so that the OWASP agent ran — can you share how the security-reviewer agent was actually run?
This never got picked up live, so here's the clean version. The demo used two distinct mechanisms, and the repo's own map labels the difference: the owasp-review skill is marked (auto) — it's project-scoped at .claude/skills/owasp-review/, and Claude Code loads it automatically into context whenever a task's description matches what the skill covers, so simply asking for a security pass over backend/src/api/ is enough to trigger it. The security-reviewer subagent (.claude/agents/security-reviewer.md) is a separate mechanism: a distinct reasoning session with its own scoped tools (Read, Grep, Glob, Bash(git diff:*)) that runs an independent second pass. Subagents like this are typically invoked either by Claude routing to them automatically when a task matches their stated purpose, or by asking for them by name — "run the security-reviewer subagent on this diff." The point of running both in the demo was that they're independent checks, not one asking the other to confirm.
Claude often proposes commands using && or |. Will Claude Code catch the follow-up commands in a chain like ls && rm -rf . separately, or does approving the first part clear the whole line?
This came in right at the end of the session and didn't get answered live — and it's really the same fragility Part 4 already covered for curl example.com, just applied to a different operator. Claude Code's documented behavior is that each subcommand in a chain (split on &&, ;, or |) is meant to be evaluated on its own — approving git status && npm test is supposed to save a rule for npm test alone, not for the whole compound string. In practice, this has been an active area of hardening: several reports against Claude Code describe cases where a chained command wasn't fully re-evaluated segment by segment. The safe takeaway matches what Part 4 already taught: don't lean on a narrow allow/ask rule to catch a compound command reliably — deny broadly, and allow only the exact single command you've verified is safe.
Managed settings get “pushed by IT.” Where does that actually go — what file, what mechanism?
The precedence got covered live (Managed beats Local beats Project beats User), but not the actual delivery mechanism, so here it is. Managed settings are OS-specific files: macOS at /Library/Application Support/ClaudeCode/managed-settings.json, Linux/WSL at /etc/claude-code/managed-settings.json, and Windows at C:\Program Files\ClaudeCode\managed-settings.json. IT typically deploys that file through the tooling they already use for fleet policy — an MDM profile on macOS (Jamf, Kandji), Group Policy or Intune on Windows — rather than a person editing it by hand on every machine. There's also a server-managed option that pushes configuration from the Claude.ai admin console directly, with no file on disk at all.
Are permissions reflected in the model's context, so it doesn't waste time trying something it isn't allowed to do?
This got a quick "no" live, plus a mention that the model gets "fine-tuned" once it hits a rejection — worth expanding on, since that phrase can land two different ways. Permission rules aren't injected into the system prompt or shown to the model up front. The agent finds out a rule exists only by attempting the action and getting a rejection back as a tool result — the same trial-and-error either way, every time. What's happening after that is in-context behavior within that same session: Claude reads the rejection like any other tool result and generally won't retry the identical blocked action again in that conversation. That's the agent reacting to feedback already sitting in its own transcript, not the underlying model being retrained on your individual denials.
Is there a suggested process for reviewing skills, plugins, and MCPs before letting engineers use them — and do Cursor or Claude Code enforce anything like AST10 before a skill reaches their marketplace?
This one got acknowledged live rather than answered in full — it's already being discussed with sec ops, and a company-wide vetting process is its own decision, separate from what one training session can settle. What today's material does give you is the closest thing to a starting method: the lethal-trifecta test from Part 3 (does the skill read private data, see untrusted content, and reach the network — all three at once?), plus the AST10 checklist's own guidance to prefer reputable sources, read a skill before installing it, pin versions, and treat adoption as its own security review. Neither Cursor nor Claude Code's marketplaces are described in today's materials as enforcing AST10 automatically — part of why this bootcamp ships its own vetted, self-contained owasp-review skill instead of pulling one from a marketplace. A formal, LVT-wide review process for engineers browsing marketplaces is still coming, separately from today.
Quiz Solutions

The seven checkpoint questions, answered.

Q1

Why is an AI first draft that looks idiomatic the most dangerous kind of code?

  • It is written faster than a human would write it
  • Idiomatic code cannot be unit-tested
  • Tidiness disarms the reviewer — you read the shape, nod, and move on ✓
  • The compiler skips its checks on idiomatic code

The shape is what the model gets right; the logic is what it gets wrong — and a clean shape talks the reviewer out of looking closely.

Q2

In /self-verify, what do you do with a green test that asserts nothing?

  • Treat it as red — strengthen the assertion and re-run ✓
  • Accept it; a passing test is a passing test
  • Delete it so the suite stays green
  • Leave it for the security hat to handle

A vacuous test green-lights whatever bugs it should have caught. It only counts once it can actually fail.

Q3

Why run three separate single-hat reviewers instead of one wearing all four hats?

  • Running them in parallel is faster
  • Each hat needs a different LLM
  • A single hat can only check formatting
  • One agent blends the conflicts away; independent hats go deep and can disagree ✓

The value isn't any one hat's output — it's the deltas between them, and the tradeoffs they surface by disagreeing.

Q4

What lets the black-box hat catch an IDOR the all-in-one reviewer misses?

  • It reviews the implementation line-by-line
  • It sees only the contract, not the code — so tidy code can't talk it out of the bug ✓
  • It runs with elevated database privileges
  • It rewrites weak tests until they pass

Reading only contracts/openapi.yaml means it can't anchor to the code's intent or excuse the gap the way a reviewer reading clean code might.

Q5

What does “authenticated ≠ authorized” mean?

  • You must authenticate twice for risky actions
  • Authorization always happens before authentication
  • Logging in proves who you are, not that you may see this row ✓
  • Only administrators are ever authorized

A valid session proves identity. It says nothing about whether that identity is allowed to see this particular record.

Q6

A skill hits the “lethal trifecta” when it can do all three of what?

  • Read private data, see untrusted content, and reach the outside ✓
  • Read code, write code, and run tests
  • Install plugins, auto-update, and log errors
  • Use the network, the disk, and the GPU

All three together is what makes a skill dangerous — any one or two alone is a much smaller risk.

Q7

In what order does Claude Code check permission rules, and which wins?

  • allow → ask → deny; the most specific rule wins
  • deny → ask → allow; the first match wins — a broad deny beats a narrow allow ✓
  • ask → allow → deny; the last match wins
  • There is no order; the AI decides from CLAUDE.md

Order beats specificity every time — which is exactly what makes a broad deny such a reliable backstop.

GitHub Actions

Run the security review yourself.

Around 7–8 people finished this live. If you didn't get to it, the guide still works post-class — same repo, same steps.

Base setup

1. Fork anshajk/claude-security to your own GitHub account.

2. Install the Claude GitHub App on your fork and grant it permission to comment on PRs.

3. Add your ANTHROPIC_API_KEY as an Actions secret in your fork's settings.

4. Enable GitHub Actions on the fork — it's off by default on a fork.

5. Create a change (the guide uses demo_file.md) and open a PR against your own fork's main branch, not the original repo — that's the one mistake the guide flags most.

Triggering the OWASP-specific review

No new setup — same fork, same app install, same secret, from the base steps above. The difference is where you put your change. The general Claude review comments on any PR, but the dedicated OWASP pass only fires when the change lands inside one of three watched paths (this is the paths: filter in the workflow file, wired to wake the job only on the vulnerable surface):

backend/src/** · frontend/** · contracts/openapi.yaml

Make a new file or edit an existing one inside one of those three paths, then open the PR the same way as before — against your own fork, not the original repo. That's the whole difference from the base setup.

Example from the session: a PR that just adds backend/src/new.md — trivial on its own, but landing inside backend/src/** is what's enough to wake the OWASP-pass job and get the automated findings posted as PR comments.
